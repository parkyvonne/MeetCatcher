# 다인원(6인) 회의 일정 조율 서비스 — 설계 리서치

## 배경

같은 회사 동료 6명이 다음 주까지 1시간짜리 회의를 잡아야 하는 상황을 가정합니다.
- 누군가는 점심 직후를 피하고 싶어함
- 어떤 사람은 특정 요일에 외근이 많음
- 필수 참석자와 선택 참석자가 혼재

이런 조건들 속에서, 사용자가 동료들의 상황을 파악하고 모두가 납득할 수 있는 시간을 찾아 회의를 확정할 수 있는 솔루션을 설계합니다.

---

## 1. 문제 정의

**어떤 문제를 발견했는가, 왜 문제라고 판단했는가**

많은 캘린더 서비스는 참석자의 상태와 업무 맥락을 회의 시간 결정에 필요한 정보로 충분히 전환하지 못합니다. 예를 들어 "누가 반드시 참석해야 하는지", "가능하지만 피하고 싶은 시간은 언제인지" 같은 조건이 대부분 **가능/불가능이라는 이분법**으로만 처리됩니다. 그 결과 형식적으로는 비어 있는 시간이더라도 실제로는 참석자 만족도가 낮거나, 이후 잦은 재조율로 이어지는 경우가 발생합니다.

일반적인 약속은 응답이 늦거나 일정이 바뀌어도 기다릴 여지가 있지만, 업무 회의는 의사결정과 후속 업무가 연결되어 있어 **제한된 시간 안에 확정하는 것**이 중요합니다. 조율이 길어지는 사이 다른 일정이 먼저 잡히거나, 회의 준비·진행이 지연되는 손실도 생깁니다.

**근거 데이터**
- Reclaim.ai 2024년 설문(1,300명 이상 직장인 대상): 주당 평균 17.1건의 회의 중 4.2건이 재조율됨
- Reclaim.ai 추가 조사: 1:1 회의만 놓고 보면 매주 42.4%가 재조율됨

이 과정에서 주최자는 캘린더와 메신저를 오가며 참석자 상황을 다시 파악하고, 우선순위를 판단하고, 결정 이유까지 설명해야 합니다. 인원이 늘어날수록 판단해야 할 조건과 예외도 함께 늘어나, **목소리가 크거나 먼저 답한 사람의 일정 위주로 회의 시간이 정해지기 쉬운 구조**가 됩니다.

> 이는 단순한 일정 조율의 불편함이 아니라, **참석자 간 맥락 정보가 손실되면서 발생하는 구조적 문제**로 판단했습니다.

---

## 2. 해결 방법

일정 조율을 "빈 시간을 찾는 과정"이 아니라, **참석자가 납득할 수 있는 시간을 정해진 시간 안에 만드는 과정**으로 재정의해 설계했습니다.

### 핵심 설계 요소

| 요소 | 내용 |
|---|---|
| 참석자 구분 | 필수 참석자 / 선택 참석자로 구분해 회의 영향도에 차등 부여 |
| 응답 단계 확장 | 가능 여부 대신 **불가능 · 피하고 싶음 · 선호**의 3단계로 확장 |
| 입력 방식 | 요일×시간 그리드에서 **드래그로 다중 선택** |
| 자동화 | 반복되는 제약·자주 피하는 시간은 **프리셋 자동 반영** |
| 캘린더 연동 | 참석자의 기존 일정은 **사전 배제** |
| 빠른 응답 | "아무 때나 괜찮아요" **원탭 버튼** 제공 |
| 마감/확정 | 마감 기한을 촉박한 옵션으로 기본 설정, 미응답자는 가능으로 간주해 **자동 확정** |
| 결과 전달 | 확정 알림에 전체 공통 사유 + 개인별 맞춤 사유(선호 미반영 시) 함께 전달 |

입력 부담이 커지면 응답률이 낮아질 수 있다는 점을 고려해, 정교함과 응답 편의성 사이의 균형을 맞추는 데 집중했습니다.

---

## 3. 설계 근거 및 검토했던 대안

**가장 크게 고민한 지점**: 정보의 정확도 vs 응답 속도의 균형. 선호 데이터를 많이 받을수록 정교한 판단이 가능하지만, 입력 과정이 길어지면 업무 회의에 필요한 "빠른 확정"과는 멀어집니다.

### 검토 후 폐기한 대안
**순차형 구조** — 필수 참석자 시간을 먼저 확정한 뒤, 선택 참석자가 그중 가능한 시간을 고르는 방식을 초기에 검토했으나, 단계를 나눌수록 전체 조율 시간이 길어지고 그 사이 다른 일정이 먼저 잡히는 문제가 있어 폐기했습니다.

### 최종 채택 방식
- 마감 기한을 고정하고, 그 시점까지 수집된 데이터를 기준으로 자동 확정
- "아무 때나 괜찮아요" 원탭 응답은 일부 선호 데이터를 포기하더라도 **응답 자체의 완료를 빠르게 만들기 위한 장치**
- 모든 참석자의 조건을 한 번에 수집 → 시스템이 회의 성립 기준과 선호를 함께 계산 → **상위 3개 후보 + 판단 근거** 제시
- 최종 결정은 주최자가 하되, 필수 참석자의 가능 여부, 선호 시간, 선택 참석자의 영향도 등을 **객관적 지표로 제시**해 특정 참석자나 주최자에게 책임이 쏠리지 않도록 설계
- 상위 후보 점수가 동일할 경우 **가장 빠른 일자**로 확정

### 설계 철학
업무 맥락에서는 모든 조건을 완벽히 만족하는 시간을 찾는 것보다, **정해진 시간 안에 참석자들이 납득할 수 있는 결정을 만드는 것**이 더 중요하다는 판단 하에 전체 흐름을 설계했습니다.
